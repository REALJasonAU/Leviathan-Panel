# Leviathan Architecture

## Goals

Leviathan is split into four major planes so the product stays scalable and easier to reason about than a tightly coupled panel-plus-daemon stack:

1. Presentation plane: the Svelte panel used by admins, staff, and end users.
2. Control plane: the Fastify API that owns auth verification, authorization, persistence, audit logs, policy, and node orchestration.
3. Execution plane: the daemon installed on Linux nodes that manages Docker containers, files, backups, metrics, and runtime hooks.
4. Contract plane: shared schemas, permission maps, and environment tooling used by all services.

## Monorepo Layout

- `apps/panel`
  - Svelte SPA
  - Firebase client auth
  - Admin and user dashboard views
  - Console, files, environment, node, template, and alert UI shells
- `apps/api`
  - Fastify REST API
  - Firebase Admin token verification
  - Firestore repositories
  - WebSocket daemon bus
  - Audit/event pipeline
- `apps/daemon`
  - Node-side service
  - Docker orchestration via Docker Engine socket
  - File workspace management
  - Metrics collection
  - Backup and task runners
  - Health endpoint
- `packages/shared`
  - Zod schemas
  - Permission registry
  - Role helpers
  - Environment parsing and validation
  - Daemon protocol message contracts

## Runtime Model

### Panel

The panel is a fast client-rendered Svelte application. It uses Firebase Authentication for sign-in and receives an ID token from Firebase. That token is sent to the API on each request. The panel never communicates directly with nodes.

### API

The API acts as the only trusted control point.

- Verifies Firebase ID tokens with Firebase Admin.
- Loads the current user, roles, permissions, and server assignments from Firestore.
- Stores nodes, templates, servers, allocations, schedules, backups, metrics, webhooks, and audit entries in Firestore collections.
- Maintains authenticated WebSocket sessions to each daemon.
- Dispatches lifecycle commands to the correct node.
- Normalizes metrics, status, and event updates from daemons back into Firestore.

### Daemon

The daemon is intentionally small and opinionated.

- Registers with a bootstrap token generated by the API.
- Exchanges the bootstrap token for a long-lived daemon token.
- Opens an authenticated WebSocket session to the API.
- Creates one workspace directory per server under a configured base path.
- Starts Docker containers with bind mounts, network settings, resource limits, startup commands, and environment variables supplied by the API.
- Reports metrics, status, and command results to the API.

## Data Ownership

- Firebase Authentication owns user identity.
- Firestore owns application state and audit history.
- The daemon owns runtime execution details on each node.
- Docker owns container isolation, process execution, and cgroup-level resource controls.

## Communication Design

### Panel to API

- HTTPS + Bearer Firebase ID token
- JSON REST for CRUD
- Server-sent real-time can be added later, but this first pass uses polling-friendly APIs and a daemon bus on the backend

### API to Daemon

- HTTPS bootstrap registration
- Authenticated WebSocket for command and event transport
- All daemon tokens are stored hashed in Firestore
- Token rotation is supported from the API

### Daemon to Docker

- Local Docker socket access
- Containers are named `leviathan-{serverId}`
- Each server workspace is bind-mounted into the container

## Environment Variable System

Templates can define structured environment metadata:

- `key`
- `displayName`
- `description`
- `defaultValue`
- `required`
- `secret`
- `readonly`
- `validationRule`
- `allowedValues`

The API can import `.env.example` content, infer defaults, and validate submitted values before provisioning a server. Secret values are masked in API responses and never echoed in audit payloads.

## Security Boundaries

- Firebase ID tokens are verified server-side only.
- Frontend receives no daemon secrets.
- Bootstrap and daemon tokens are hashed at rest.
- Permission checks are executed in the API for every sensitive action.
- The daemon only accepts commands over the authenticated API channel.
- Environment variables marked secret are redacted in logs and API payloads.

## Extensibility

The control plane emits domain events at key points:

- node registered
- server created
- server power action
- environment changed
- backup completed
- alert triggered

These hooks support future Discord bots, billing bridges, marketplace plugins, and outbound webhooks without requiring a rewrite of core flows.
